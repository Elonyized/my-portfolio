PDO Best Practices Every PHP Developer Should Know

June 10, 2026 8 min read Sule ThankGod

PDO (PHP Data Objects) is the right way to talk to databases in PHP. Yet most tutorials cover only the surface — basic prepare() and execute() calls — and leave out the details that separate a secure, maintainable data layer from one that quietly causes pain for months.

This post covers the things I wish I had known when I started: proper error modes, explicit binding types, persistent connections, and the patterns that make PDO a genuine pleasure to work with.

1. Always set the error mode

By default PDO silently swallows errors, which makes debugging a nightmare. The first thing you should do after creating a connection is switch to exception mode:

$pdo = new PDO($dsn, $user, $pass, [
    PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
    PDO::ATTR_EMULATE_PREPARES   => false,
]);

Setting ATTR_EMULATE_PREPARES to false forces PDO to use true prepared statements on the database server instead of client-side emulation — this gives you real parameterisation and real type safety.

2. Bind with explicit types

When using LIMIT and OFFSET, PDO can't infer the integer type from PHP's loosely typed system. Bind explicitly:

$stmt = $pdo->prepare('SELECT * FROM property LIMIT :limit OFFSET :offset');
$stmt->bindValue(':limit',  $limit,  PDO::PARAM_INT);
$stmt->bindValue(':offset', $offset, PDO::PARAM_INT);
$stmt->execute();

Skipping this causes MySQL to receive the integers as strings, which breaks the query silently in older MySQL versions and can cause surprising results in newer ones.

3. Wrap your connection in a singleton

Opening a new PDO connection on every page request is wasteful. A simple singleton ensures a single connection per process:

class DB {
    private static ?PDO $instance = null;

    public static function conn(): PDO {
        if (self::$instance === null) {
            $dsn = 'mysql:host=' . DB_HOST . ';dbname=' . DB_NAME . ';charset=utf8mb4';
            self::$instance = new PDO($dsn, DB_USER, DB_PASS, [
                PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
                PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
                PDO::ATTR_EMULATE_PREPARES   => false,
            ]);
        }
        return self::$instance;
    }
}

4. Keep SQL in class methods, not in page files

Your page files (views) should never contain raw SQL. Put all database logic in a dedicated class method. This makes queries testable, searchable, and easy to optimise later without hunting through templates.

Key takeaways

  • Always use ERRMODE_EXCEPTION — silent failures cost you hours of debugging.
  • Disable prepared-statement emulation for real parameterisation.
  • Bind integers with PDO::PARAM_INT — especially for LIMIT and OFFSET.
  • Centralise your connection in a singleton class.
  • SQL belongs in model/repository classes, not in view files.
