Why Most Design Systems Fail (and What to Do Instead)

May 28, 2026 7 min read Sule ThankGod

Most design systems get built once, celebrated in a launch announcement, and then quietly ignored within six months. Engineers route around the components because they're "easier to just rebuild," and designers stop referencing the library because it never quite matches what they need. I've seen this happen twice from the inside before I understood why.

The third attempt — what became Bloom, a 180-component system now used by four product teams — survived because we changed almost nothing about the components and almost everything about how the system was built and maintained.

1. The system was designed by committee, for nobody

The first version tried to satisfy every team's preferences simultaneously. The result was a button component with fourteen variants, most of which existed because one designer once asked for them. Nobody used more than three. Optionality felt generous but actually made the system harder to learn and easier to ignore.

Bloom shipped with one button, one input, one card — each with a small, deliberate set of states. Adding a variant required a real use case, not a hypothetical one.

2. Documentation was an afterthought

A component without usage guidance is just a component nobody trusts. The second attempt had a Figma file and nothing else — no prop tables, no do/don't examples, no rationale. Engineers guessed at intent and got it wrong constantly, which eroded confidence in the whole system.

Every Bloom component ships with three things: a live coded example, a written rationale for why it looks the way it does, and explicit anti-patterns — what not to do and why.

A design system isn't a component library. It's a set of agreements, and agreements only work if people understand why they exist.

3. There was no owner after launch

Design systems decay the moment nobody is accountable for them. The first two attempts had a "launch team" that disbanded after shipping v1. Six months later nobody knew who could approve a new component, so teams just built their own instead.

Bloom has a standing two-person team with dedicated time each sprint. Every new component request goes through a short, public review — not because we love process, but because an unowned system is a dead system within a year.

4. It optimised for completeness instead of adoption

It's tempting to build every component you can imagine before calling something v1. That's backwards. We shipped Bloom with eleven components — covering 80% of real screens already in production — and grew it based on actual requests from teams trying to use it, not speculative ones.

What actually made teams adopt it

  • A small, opinionated core instead of an exhaustive, optional one
  • Documentation that explains why, not just how
  • A named, funded owner — not a side project
  • Growth driven by real usage, not anticipated need
  • Migration support — we helped teams adopt it instead of just announcing it existed

Key takeaways

  • Fewer, better-defined components beat exhaustive variant coverage.
  • Document the reasoning, not just the API.
  • A design system needs an ongoing owner, not a launch team.
  • Ship a small core early and grow it from real demand.
